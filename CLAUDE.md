# CLAUDE.md

Hamster is a self hosted, S3 compatible object store in a single Go binary, built around erasure coded durability and first class versioning and object lock.

**Positioning:** the target user runs real workloads with compliance-shaped needs — retention, object lock, audits (HIPAA, SEC 17a-4 territory) — without a platform team. Design and performance work should serve that user: use the hardware well and be plenty fast, but never trade durability, compliance correctness, or operational simplicity for benchmark wins. Small-file microbenchmark supremacy is explicitly not a goal. In public-facing docs, name MinIO's community-edition archive as the factual origin story. Naming other open-source stores is allowed where it serves the reader — chiefly the head-to-head comparison in [docs/COMPARISON.md](docs/COMPARISON.md) — provided it stays factual, neutral, and never disparaging, names where Hamster is itself behind, and is dated (the field moves). Outside that deliberate comparison, prefer to generalize rather than name rivals in passing.

> **Status: early development (v0). Not production ready.** Nothing described in the docs is a guarantee yet; it is the design being built. Keep all documentation honest about this — features are goals until they ship and survive the simulation harness.

## Critical invariants — never violate these

These are the load bearing design decisions. Code or docs that break them are wrong even if tests pass.

1. **Object data never passes through the Raft log.** Erasure coded shards are written directly to storage nodes. Only the small metadata commit (key, version, size, checksums, shard locations) goes through Raft. Durability comes from the EC spread, not from consensus. See [ADR-0005](docs/adr/0005-metadata-badgerdb-raft.md).
2. **All on disk and on wire formats are additively versioned protobuf.** Every persistent or networked structure carries a version field. Fields are only ever added — never removed, never repurposed. New code must always read old formats. See [ADR-0008](docs/adr/0008-versioned-formats-rolling-upgrades.md).
3. **Metadata models every key as an ordered list of versions**, never a single record per key — even when versioning is disabled and the list holds one entry. Enabling versioning must never require a schema migration. See [ADR-0006](docs/adr/0006-versioning-and-object-lock.md).
4. **COMPLIANCE mode object lock has no override path.** Not for administrators, not for operators, not behind a flag. If a code path can delete or shorten retention on a COMPLIANCE locked version, it is a bug. GOVERNANCE mode is the bypassable one; COMPLIANCE is not.
5. **Durability and consistency changes must pass the deterministic simulation harness.** Any change touching the write path, repair, placement, Raft, or formats must survive the simulated failure schedules before it merges. See [ADR-0009](docs/adr/0009-deterministic-simulation-testing.md).
6. **Objects are immutable blobs.** Written once, never edited in place. Overwrites create a new version; the current version pointer resolves inside the metadata transaction.
7. **The partition count is fixed and never resized.** Rebalancing migrates partitions between nodes; it never re-encodes objects. See [ADR-0004](docs/adr/0004-partitioned-placement.md).
8. **The failure domain is the node, not the disk.** Shard placement must never put two shards of one object on the same node. Hosts and zones group nodes above that ([ADR-0016](docs/adr/0016-failure-domain-hierarchy.md)); spreading across them is a placement objective — the node rule is the hard floor.

## Go conventions

- Standard Go style: `gofmt`, `go vet`, idiomatic naming. Exported identifiers get doc comments.
- Errors are wrapped with context (`fmt.Errorf("...: %w", err)`) and checked, not ignored.
- No global mutable state; pass dependencies explicitly so the simulation harness can substitute clocks, networks, and disks.
- Anything that touches time, randomness, the network, or the filesystem goes behind an interface that the simulator can control. Determinism is a feature, not a test affordance bolted on later.
- Version IDs are UUIDv7, minted in-repo by `meta.NewVersionID` from explicit clock and PRNG inputs — `google/uuid`'s `NewV7` reads ambient time and randomness, which the determinism rule forbids. Kept as `[16]byte` (`meta.VersionID`). See [ADR-0007](docs/adr/0007-uuidv7-version-ids.md).
- Prefer the standard library. Every dependency must justify itself — the single binary, no external services promise extends to keeping the module graph small.
- **No cgo, anywhere.** Hamster builds with `CGO_ENABLED=0`, always. Dependencies must be pure Go — a library that requires cgo is disqualified no matter how good it is. (Go assembly is fine: that is how `klauspost/reedsolomon` gets its SIMD speed.) This is what keeps the binary truly static and cross-compilation trivial; see [ADR-0002](docs/adr/0002-single-binary-no-external-dependencies.md).
- **Dependency licensing: permissive only.** Any imported package must be Apache 2.0 or similarly permissive (MIT, BSD, ISC), including transitive dependencies. No copyleft (GPL, LGPL, MPL) and no source-available licenses (BUSL, SSPL). Exceptions require an ADR. See [ADR-0011](docs/adr/0011-permissive-only-dependencies.md).

## Build, test, and lint

Targets live in `Taskfile.yml` ([Task](https://taskfile.dev), not make):

```sh
task build      # CGO_ENABLED=0 go build ./...
task test       # all tests, including the simulation harness
task test-race  # under the race detector, excluding the real-process internal/cluster suite (the detector needs cgo; race builds are never shipped)
task test-race-cluster  # race-check internal/cluster on its own — best-effort: loopback-mTLS timing under the detector can flake without a data race
task check      # go vet + gofmt
task e2e        # end-to-end suite over the real binary: real processes, real CLI, loopback mTLS (test/e2e, `e2e` build tag)
task compat     # third-party client compatibility suite (aws CLI, rclone, restic, s3cmd) against an in-process gateway; absent tools skip
task dist       # cross-compile the static, version-stamped release binaries + SHA256SUMS into dist/ (VERSION=vX.Y.Z)
task release    # run the full gate, cross-compile, tag, and publish a GitHub release (VERSION=… TITLE=… NOTES=…)
```

`build`, `test`, `test-race`, and `check` must pass before any commit. `test-race` deliberately omits `internal/cluster`: that suite is real processes over real loopback mTLS — the design's best-effort plumbing layer (SIMULATION.md) — and per package the detector finds no races there, but the whole set under the detector's ~10x slowdown manufactures timing flakes (join-token TTLs lapsing, leadership races) that are load artifacts, not data races. The kept packages cover where a real race would live: the simulated core, the data plane, and the `sys` adapters. When touching `internal/cluster`, also run `test-race-cluster` (best-effort) — a data race there is a real failure; a one-off timing flake is not. `e2e` and `compat` are not part of the pre-commit gate (`e2e` builds and spawns real processes — hermetic but slower; `compat` shells out to locally installed third-party binaries — not hermetic). Run `e2e` for changes touching `cmd/hamster` or `internal/cluster`, and `compat` for changes touching `internal/gateway` or `internal/sigv4`. Its tests live in `test/compat` behind the `compat` build tag.

## Code layout

- [`internal/seam`](internal/seam/) — the interfaces between core logic and the world: `Loop`, `Clock`, `Transport`, `MessageHandler`, `Disk`. Core code receives these; it never touches the OS directly. Randomness is a seeded `*math/rand/v2.Rand`, no interface needed.
- [`internal/sim`](internal/sim/) — the deterministic simulation harness ([ADR-0009](docs/adr/0009-deterministic-simulation-testing.md)): global event queue, virtual time, seeded PRNG, faulty network, crash-faithful disk. New core logic gets built and tested under this from day one — the single-node store composition (meta + blob + wal) already runs under adversarial crash schedules with a model checker.
- [`internal/meta`](internal/meta/) — the metadata model ([METADATA.md](docs/METADATA.md), [ADR-0014](docs/adr/0014-metadata-keyspace-design.md)): the version-list keyspace, deterministic apply, version IDs, the stored cluster-layout singleton with its compare-and-set install ([ADR-0028](docs/adr/0028-stored-cluster-layout.md)), the replicated per-member node registry (`NodeRecord` under `s/node/`, carrying each member's failure-domain labels, capacity weight, operator-set drain flag — `SetNodeDraining` — replacement pairing — `SetNodeReplacedBy` ([ADR-0004](docs/adr/0004-partitioned-placement.md)) — so any leader composes the layout, not only the join issuer — and the fingerprint of the CA that signed its current leaf — `SetNodeLeafCA`, so CA rotation counts who is still on the old CA ([ADR-0033](docs/adr/0033-ca-rotation.md)) — and its advertised binary version and declared protocol **generation** — `SetNodeVersion`, the leader's version monitor keeping each member's record current so the cluster's effective generation (the min across live members) rolls etcd-style as the last node upgrades ([ADR-0034](docs/adr/0034-rolling-upgrade-machinery.md))), the replicated CA **trust bundle** singleton (`TrustBundle` under `s/trust`, the generational set of trusted CA *certificates* and the issuing CA, compare-and-set installed — CA rotation's dual-trust basis, certs only, never the key — [ADR-0033](docs/adr/0033-ca-rotation.md)), each version's additive encryption-at-rest fields (`EncAlgorithm` + `WrappedDEK`, the per-object DEK wrapped under the cluster KEK — [ADR-0021](docs/adr/0021-envelope-encryption-at-rest.md); `ApplyReEncodeObject` preserves them since re-encode re-shards ciphertext, plus `KEKFingerprint` naming the wrapping key and the `EncryptionPosture`'s current/rotating-to fingerprints for master-key rotation — [ADR-0032](docs/adr/0032-master-key-rotation.md): the `BeginKEKRotation`/`RewrapDEK`/`CompleteKEKRotation` proposals, `RewrapDEK` touching only the wrapped key so a COMPLIANCE-locked version rewraps with lock and bytes intact), and the hand-written protobuf codecs for records and Raft-log proposals ([ADR-0023](docs/adr/0023-handwritten-protowire-codecs.md)). Stdlib plus `protowire` only, no seam imports — proposals carry every input apply needs. Tested against an independent reference model with seeded randomized workloads, each run proving byte-identical restart equivalence through the `Persister` seam.
- [`internal/raftnode`](internal/raftnode/) — etcd-io/raft's `RawNode` driven over the seam ([ADR-0012](docs/adr/0012-etcd-raft-consensus-library.md)): WAL-backed storage with snapshot compaction (crash-safe single-frame log rotation; lagging followers catch up via streamed installs), transport glue, apply into `meta.Store` made durable in a per-replica BadgerDB store through the `Persister` seam ([ADR-0005](docs/adr/0005-metadata-badgerdb-raft.md)): in production that store is the boot source of truth — loaded with the applied index it carries, only the un-applied WAL tail replayed — with the WAL (whose snapshots are full dumps) as the rebuild fallback if it is lost or corrupt; under the simulator a crash-faithful disk-backed stand-in exercises the persister path (load, skip-don't-reapply, snapshot-install reset, WAL rebuild) while the log-only mode proves the fallback, membership (learner joins via `AddNode`, the five-voter cap with automatic caught-up promotion per [ADR-0017](docs/adr/0017-raft-voter-cap-learners.md), a replicated address book in conf-change contexts and snapshots, and `RemoveNode` eviction with a replicated removed-ID tombstone — an evicted ID is never re-admitted, so the joiner-driven admit path cannot take a removed node back even if it keeps running or restarts; a returning node needs a fresh join with a new ID, and an evicted node self-stops via the `OnRemoved` callback), and the election timer Hamster owns for determinism ([ADR-0024](docs/adr/0024-deterministic-election-timing.md)). Proven by cluster schedules under the simulation harness; not yet wired into `hamster serve` (single node until v0.2 ships).
- [`internal/sigv4`](internal/sigv4/) — AWS Signature Version 4 verification ([ADR-0018](docs/adr/0018-sigv4-auth.md)): Authorization-header, presigned-query, unsigned-payload, and aws-chunked streaming modes. Pure stdlib; time is a parameter (`Verify(r, now)`), never an ambient clock read. Conformance-tested against AWS's published example signatures.
- [`internal/gateway`](internal/gateway/) — the S3 HTTP front end ([S3-API.md](docs/S3-API.md)): SigV4 wiring, path-style and virtual-hosted routing, XML envelopes, and handlers that turn requests into metadata proposals and blob I/O. The store and rng are loop-owned; handlers post to the loop and wait. The SSE-S3 surface ([ADR-0021](docs/adr/0021-envelope-encryption-at-rest.md), `sse.go`): `x-amz-server-side-encryption: AES256` is echoed on PUT/GET/HEAD for an encrypted version (read from its own record), the request header is validated against the cluster posture (`Config.EncryptionEnabled`) — an `AES256` request the server cannot honor is refused rather than silently storing plaintext — and SSE-KMS/SSE-C are refused honestly. Tested end to end with an independently written client-side signer.
- [`internal/wal`](internal/wal/) — append-only record logs on a `seam.Disk`: framed, checksummed, torn-tail-tolerant. Carries `RowLog`, the `meta.Persister` for stores living on a `Disk` (the simulation harness today, the Raft log's foundation in v0.2). Record payloads are versioned protobuf; the framing is a container, not a format.
- [`internal/stream`](internal/stream/) — the framed object stream ([DATA-STREAM.md](docs/DATA-STREAM.md)) between the gateway and erasure coding: fixed-size chunks, a header plus trailer index for random access (a Range read touches only the chunks it covers), per-chunk CRC-32C verified on every read. Ships identity frames and **encrypted frames** (per-chunk AES-256-GCM under a caller-supplied per-object DEK, the chunk index as nonce, the GCM tag joining the per-chunk CRC — v0.7, [ADR-0021](docs/adr/0021-envelope-encryption-at-rest.md)); compression stays a reserved header flag, refused until its code exists. Pure stdlib, pure computation — no clock or I/O, so no seam (the DEK is a caller input, not minted here). Golden-pinned encoding for both frame shapes; every single-byte corruption and truncation proven detected, wrong-key and tamper rejected by the GCM tag.
- [`internal/keys`](internal/keys/) — envelope-encryption key handling ([ADR-0021](docs/adr/0021-envelope-encryption-at-rest.md)): per-object data-encryption keys (`DEK`, minted from an injected entropy `io.Reader` — `crypto/rand` in production, a seeded reader under the simulator, so the DEK is the only random input to the encrypted write path) and the symmetric cluster `KEK` that wraps them (stdlib AES-256-GCM; the wrap nonce is caller-supplied and derived from the object's unique version ID, so no random-nonce reuse bound applies and the wrap is deterministic). `LoadKEK` parses raw/hex/base64 material; the KEK lives only in memory and refuses to wrap/unwrap when unloaded. One source today — `--master-key-file`, behind a "source returns a 32-byte KEK" shape so passphrase/argon2id and `--master-key-command` (Vault/AWS-Secrets) are additive later. Each loaded KEK carries a content **fingerprint** (truncated HMAC over a fixed label — a key-check value, preimage-resistant, identifying a key without revealing it), the identifier master-key rotation hangs its observability on ([ADR-0032](docs/adr/0032-master-key-rotation.md)). Pure given its inputs — key-file *reading* is the boot layer's job, so no seam here. Wrong-key and tamper rejected by the GCM tag; an end-to-end test mints→wraps→unwraps and decrypts a real `stream` frame.
- [`internal/ec`](internal/ec/) — the erasure-coding engine ([ERASURE-CODING.md](docs/ERASURE-CODING.md), [ADR-0026](docs/adr/0026-stripe-and-shard-layout.md)): the storage profiles and auto ladder ([ADR-0015](docs/adr/0015-storage-profiles.md)), the small-object k=1 rule, the stripe codec turning a framed stream into k+m self-describing shard files (`klauspost/reedsolomon` per [ADR-0013](docs/adr/0013-klauspost-reedsolomon.md)), random-access reads that reconstruct from any k, and checksum-verified shard rebuild for repair. Pure computation over caller-supplied readers/writers — no seam. Round trips proven under every survivable loss pattern of every profile; golden-pinned shard header.
- [`internal/place`](internal/place/) — the placement function ([ADR-0027](docs/adr/0027-v03-distributed-data-path.md)): data ID → partition (mixed FNV-1a 64, golden-pinned — partition values live in committed records forever), and partition → ordered node list. The member set comes from a stored, versioned `ClusterLayout` ([ADR-0028](docs/adr/0028-stored-cluster-layout.md)) read once per operation as a `place.Layout` snapshot (v0.4 pass 1). Placement is failure-domain-aware (v0.4 pass 2, [ADR-0016](docs/adr/0016-failure-domain-hierarchy.md)): a greedy selection over the rendezvous ranking spreads shards across zones, then hosts, then nodes — node-distinct by construction (the hard floor), prefix-stable (narrow widths a prefix of wide ones), and collapsing to the bare rendezvous ranking on a single-host/single-zone cluster. Capacity weighting (v0.4 pass 3, [ADR-0004](docs/adr/0004-partitioned-placement.md)) biases the ranking by each node's relative weight so a larger node holds proportionally more partitions within the spread — deterministic integer weighted rendezvous (the log method in fixed point, never floats, so every node computes the identical mapping), equal weights leaving placement byte-for-byte unchanged. Draining (v0.4, [ADR-0004](docs/adr/0004-partitioned-placement.md)) demotes a node below every active one in the greedy selection, so new writes steer around it (it is still chosen to fill width as a last resort — durability is never traded for avoidance). Because shard addressing is positional and derived from the member set, a layout change *relocates* where a shard lives; `Layout.Locate` resolves both the new placement and the prior one (`Previous`, set during a transition) so a reader can dual-read each shard from wherever it currently sits — the foundation that makes any layout change safe for existing data. Transition tracking is in progress: the dual-read primitive and GET consuming it have landed; installing the transition on a membership change and migrating shards old→new (repair) are the remaining v0.4 work ([ADR-0004](docs/adr/0004-partitioned-placement.md)) — until they land, a drain/weight change is only safe on a cluster with no data at risk. Pure computation, no seam.
- [`internal/datapath`](internal/datapath/) — shard bytes between nodes ([ADR-0027](docs/adr/0027-v03-distributed-data-path.md)): the channel envelope (Raft and data traffic share the transport), versioned protobuf transfer messages (golden-pinned), the server staging incoming shards on a `seam.Disk` (durable + complete = the `.ok` commit marker, synced strictly after the bytes; checksum verified against what metadata will record), and the client state machines (`WriteStream` windowed with timer retransmission, `Fetch`, `Delete`). Reliability rules: chunks idempotent by (stream, key, offset); acks cumulative and stream-echoed; a receiver with no staging for a mid-stream message answers with silence — any stateful answer is ambiguous under reordering, so receiver crashes fail by timeout, never falsely. Loop-owned; proven under simulated drops/duplicates/reorderings/crashes.
- [`internal/coord`](internal/coord/) — the data-path coordinator ([ADR-0027](docs/adr/0027-v03-distributed-data-path.md)): loop-owned state machines turning one S3 operation into placement (resolved per operation from the stored `ClusterLayout`, [ADR-0028](docs/adr/0028-stored-cluster-layout.md)) + erasure coding + shard transfer + a Raft metadata commit. The PUT paces the body through the pure stream→ec composition into datapath write streams (window-bounded bursts, never blocking the loop), enforces the [ADR-0015](docs/adr/0015-storage-profiles.md) ack rule (all k+m healthy, k+1 floor, refuse below with `ErrRefused`), and proposes PutObject only after the shards are durable — the linearization point, the only bytes that touch the Raft log. When the cluster's encryption posture is on ([ADR-0021](docs/adr/0021-envelope-encryption-at-rest.md), resolved per write via `Config.Encryption`), the PUT mints a per-object DEK from `Config.Entropy` (`crypto/rand` in production, a seeded reader under the sim — the DEK is the encrypted path's only random input), encrypts through the stream transform, wraps the DEK under the node's KEK (nonce = the version ID), and commits the wrapped DEK; GET unwraps from the version's own record and decrypts, refusing loudly if the KEK is unavailable. Reads are posture-free (each version records whether it is encrypted *and* the fingerprint of the KEK that wrapped its DEK, so a GET resolves the right key through the keyring even after a rotation), so a cluster holds a mix; repair/scrub/re-encode stay key-free on ciphertext. Master-key rotation ([ADR-0032](docs/adr/0032-master-key-rotation.md)) is `RewrapSweep`: a leader-only, metadata-only pass that unwraps each encrypted version's DEK under the old KEK and rewraps it under the new one (`rewrap.go`, under the shared single-flight guard), committing only the small `WrappedDEK` — the shards never move, so it is COMPLIANCE-safe; the count of versions still on the old fingerprint is the provable convergence signal that closes the rotation, and the write path stamps and guards the cluster's current fingerprint (a node holding the wrong master key is refused). A passive per-node liveness detector (loop-owned, deterministic — fed by every data-plane outcome: PUT write streams, GET fetches, and repair verifies, keyed on `datapath.ErrUnreachable` so a timeout marks down but a holder that merely lacks a shard does not) lets the PUT skip a known-down node up front instead of paying its retransmit timeout — but only while enough nodes stay up to meet the floor, so the skip changes latency, never durability (repair rebuilds the skipped shard). `Coordinator.DownNodes` exposes the view for `cluster status`. The GET fetches headers from all shards, prefetches the covering slice ranges (`stream.Cover` says exactly which frame bytes a range read touches; `ec.ReadHeader` gives the recorded geometry), and decodes through the pure verifying readers — any k shards suffice, reconstruction included. During a layout transition it dual-reads each shard from its new or old home (`place.Locate`), so a rebalance never makes existing objects unreadable; in steady state this is the single placement, unchanged. `RepairSweep` is scrub and rebuild as one pass: every shard of every version hashed by its holder (the datapath `Verify` message) and compared against the entry's replicated checksums, missing/rotted shards rebuilt from any k verified survivors via `ec.Reconstruct` (corruption never laundered), rotted holders deleted before the rewrite, rebuilt shards committed through the ordinary staged-then-marker path. During a layout transition the same sweep also *migrates*: it resolves `place.Locate` (new + old home) and carries each shard to its new home — a copy, not a reconstruct (the shard sits intact at its old home, so it is fetched whole from `old[i]` and written to `new[i]`; EC reconstruction is the fallback only for a shard damaged mid-move, clean at neither home), and a sweep that migrates nothing is the convergence signal that closes the transition. And it *re-encodes* for a downsize ([ADR-0031](docs/adr/0031-reencode-across-profiles.md)): an object whose width no longer fits the active (non-draining) node count is rewritten to the active-count profile via `coord.ReEncode` (`ec`-decode the framed stream, `ec`-encode at the new k+m, write the narrower shards, commit `meta.ReEncodeObject`, drop the old) — a physical re-representation, COMPLIANCE-safe (the bytes and lock are unchanged; old shards drop only after the new are durable and committed), so a cluster can shrink below an object's width. The same step runs the other way for growth: `Coordinator.Optimize` is a sweep that *widens* every under-width object up to the active profile (the `cluster optimize` data step) — explicit, never automatic, since growth keeps data readable at its old width and re-spreading it across the new nodes is the operator's call, not a side effect of every repair sweep. A continuous background scrubber (`StartScrub`, `scrub.go`) runs the correctness sweep on its own — loop-owned and seam-clock-paced, so it runs identically under the simulator: it walks the keyspace one object per `Pace`, healing bitrot and lost shards before any read trips over them, only as the leader and only in steady state (it stands aside while a transition is open). A shared single-flight guard (`Coordinator.sweeping`) serializes every sweep — operator optimize, transition migration, and the scrubber — and the scrubber yields the guard between objects so the other two slot in within one object's work. The active-count profile also governs new writes (`put.go`), so a drain that crosses a ladder boundary steers new and existing data to the same target. Proven by simulated cluster schedules: full Raft metadata plane + data plane per node, durability checked by decoding shard files off the surviving disks and by network GETs through crashed holders; repair schedules cover an emptied node, two-shard bitrot healed without any read, beyond-tolerance honestly reported, crash-mid-sweep convergence, a transition migrated shard-by-shard to a new placement that still reads with the drained node crashed, a downsize re-encoding 4+2→3+2 that reads at the new profile with the drained node crashed, and an upsize Optimize widening 2+1→4+2 after growth that then survives two crashed nodes.
- [`internal/blob`](internal/blob/) — the v0.1 single-node data path: blobs on a `seam.Disk`, addressed by `DataID`, streamed through the write buffer (bounded memory, appended then synced) and durable before the metadata commit. Replaced by the erasure-coded path behind the gateway's `BlobStore` interface.
- [`internal/certs`](internal/certs/) — the cluster CA and the node certificates it issues ([ADR-0022](docs/adr/0022-cluster-mtls.md)): Ed25519, stdlib x509, the node ID bound as common name and DNS SAN — the identity the transport authenticates. Each CA carries a random serial and a Subject Key ID so two same-named CAs (the old and new during a rotation) are never confused; `CAFingerprint` identifies a CA cert by content and `PoolFromCAs` builds the multi-CA trust pool a dual-trust rollover needs ([ADR-0033](docs/adr/0033-ca-rotation.md)). Validity windows take time as a parameter; keys are crypto/rand by nature.
- [`internal/cluster`](internal/cluster/) — the operational cluster layer ([ADR-0022](docs/adr/0022-cluster-mtls.md)): the on-disk node identity under `<data-dir>/cluster`, join tokens (single-use, TTL-bound, pinning the CA hash so a joiner can authenticate the cluster it has no trust store for yet), the join/status wire protocol (which shares the peer transport's port, split off by ALPN — one cluster listen address per node, [ADR-0030](docs/adr/0030-single-cluster-port.md)), and the composition that runs a v0.2 metadata-plane node — raftnode over the `sys` adapters with the mTLS transport. `cluster recover` is the quorum-loss exit ([ADR-0025](docs/adr/0025-force-new-cluster-recovery.md)): offline, local-log-wins, single voter. Every node also runs the data plane behind the [ADR-0027](docs/adr/0027-v03-distributed-data-path.md) channel envelope (Raft and shard traffic share the mTLS transport) and a coordinator whose member set and auto-ladder profile come from the stored, versioned `ClusterLayout` ([ADR-0028](docs/adr/0028-stored-cluster-layout.md)); the leader reconciles that layout toward Raft membership, composing it from the replicated node registry (`meta.NodeRecord`) the issuer populates at admission — labeling each member with its failure-domain host/zone ([ADR-0016](docs/adr/0016-failure-domain-hierarchy.md): `host` auto-detected, `-zone` operator-set, captured at join) and capacity weight, so any leader (not just the issuer) builds a complete layout and placement spreads shards across failure domains. A *subtractive* change (a node draining out) opens a layout transition rather than swapping outright (`reconcileLayout`): the leader's repair sweep migrates the node's shards to the rest of the cluster and `driveTransitionClose` drops `Previous` once a sweep converges — one transition at a time (`cluster drain` refuses a second concurrent drain), additive changes (joins, weight) still riding rebuild-from-k so formation never serializes. When the drain shrinks the cluster past a storage-profile boundary the same sweep re-encodes the data down instead ([ADR-0031](docs/adr/0031-reencode-across-profiles.md)); `cluster drain` detects that, prints the before/after profile and the real per-step trade (6→5 keeps two-failure tolerance and costs only efficiency; 5→4 drops tolerance to one), and asks `[y/N]` (or takes `-reencode`). `cluster remove` then evicts a drained node for good: leader-only, gated so durability is never traded — the node must be drained and *empty* (the remaining nodes still hold every stored object at its full width, since k is never downgraded in place), then it is dropped from Raft membership and its ID tombstoned so it cannot re-admit (a return needs a fresh join). Joining a fresh node with `-replaces <old>` swaps it for an existing member at *constant* cluster size (`meta.NodeRecord.ReplacedBy`, set by the issuer as part of the join): reconcile drops the old one and adds the new in one same-size transition — the storage profile never changes (a pure migrate, not a re-encode), so it works even on a tight cluster where every node is load-bearing and a drain/remove could not make room; the old node is migrated off and evicted automatically. The intent rides the join itself (no separate command), so the swap is atomic — the new node is never counted as growth. Every node runs the continuous background scrubber (`coord.StartScrub`, leader-gated), so bitrot and lost shards self-heal without an operator. After growing a cluster, `cluster optimize` re-encodes existing data *up* to the now-larger active profile (`Coordinator.Optimize`, leader-only), spreading objects written when the cluster was smaller across the nodes added since — explicit, never automatic. It gates on layout readiness (`layoutSettled`: the committed layout equals the set Raft membership + the registry imply, no transition open) so it never widens against a stale node count: a fresh join that has not reconciled yet is a *retryable* refusal the command waits out, while an operator's drain/replace in flight is a hard refusal to resolve first. drain, replace, and any layout transition are mutually exclusive — one layout change at a time (`layoutOpInProgress`). `cluster status` reports the achieved host/zone topology, the encryption-at-rest posture, and each member's STATE: `draining` (a committed fact from the layout — the operator is removing the node) or `down` (the answering node's local, best-effort liveness view from `Coordinator.DownNodes`, taking precedence as the more urgent), else `up`. Each member also advertises its binary version and declared protocol **generation** ([ADR-0034](docs/adr/0034-rolling-upgrade-machinery.md)): supplied at startup (`WithVersion`, never persisted — the binary owns it), the leader's `versionMonitor` polls each peer's own version over the status channel and replicates it into that member's `NodeRecord` (`SetNodeVersion`), so an in-place upgrade rolls the cluster forward with no re-join. `cluster status` shows the per-node version and the **effective generation** — the min across live members, etcd-style — which advances only once the last node upgrades, plus a one-step skew note (no proposal forwarding, so a follower's new version is learned by the leader's poll, not self-proposed). `cluster can-stop <node>` is the advisory health interlock ([ADR-0034](docs/adr/0034-rolling-upgrade-machinery.md)): it answers whether taking a node down for maintenance/upgrade is safe right now — Raft quorum survives without it (a voter only when a majority remains), no *other* node is down, and no layout transition is open — exiting 0 (safe) or 1 (not) so a roll script can gate on it; it never stops a node (a node must always be stoppable in an emergency). The binary swap itself is out of band, one node at a time, however the operator runs Hamster (docker/k8s/package) — Hamster owns the safety machinery and the proof, not the swap ([UPGRADES.md](docs/UPGRADES.md)). Encryption at rest ([ADR-0021](docs/adr/0021-envelope-encryption-at-rest.md)) is a replicated posture singleton (`meta.EncryptionPosture`, enable-only): `cluster encrypt` turns it on (leader-only, refused if the leader holds no key), and each node loads its KEK at boot from `cluster run -master-key-file` — held in memory only, never persisted or sent over the wire, wired into the coordinator's `Encryption`/`Entropy`; a node whose posture is on but whose KEK never loaded refuses encrypted work loudly. Inter-node mTLS trust is the replicated CA **trust bundle** ([ADR-0033](docs/adr/0033-ca-rotation.md)), not a single static CA: the leader seeds it from the founding CA at formation, every node builds its trust pool from it (`refreshTrust`, persisted to ca.pem atomically so restarts trust the current set) and the transport reads that pool — and the node's current leaf — per handshake, so a CA rotation takes effect with no restart. `cluster rotate-ca` drives a dual-trust rollover: mint a new CA, widen the bundle to trust old+new, reissue every member's certificate onto the new CA (a leader-driven push the member validates against the bundle's issuing CA, then goes live and persists), then drop the old CA once `meta.NodeRecord.LeafCAFingerprint` shows every member reissued — the count still on the old CA is the provable convergence signal (`cluster status` shows it). The driver rides out a leadership blip (`proposeAsLeader`) and a re-run converges from any partial state; planned rotation and lost-CA-key recovery are one flow, since rotation never needs the *old* CA key. `ServeS3` puts the gateway on a node: reads from the local replica, mutations as Raft proposals (leader-only in v0.3 — no proposal forwarding yet; non-leaders answer SlowDown), objects erasure-coded through the coordinator, multipart/copy refused on this path until their EC design lands. Tested as a real multi-node cluster over loopback TLS, including the six-node S3 e2e that kills nodes mid-workload.
- [`internal/metrics`](internal/metrics/) — the hand-rolled metrics registry ([ADR-0035](docs/adr/0035-observability-metrics.md)): the one in-process source of truth for a node's signals (counters, gauges, scrape-time collector callbacks for gauges derived from live state), with a golden-pinned Prometheus text exposition. Pure — no clock, no I/O, no seam imports; a caller computes a duration through the seam clock and records the value, so recording stays deterministic, and a metric is never an input to replicated state. `Snapshot` gathers the registry into typed `Family`/`Sample` values — the one gather the Prometheus text (`RenderText`) and the hand-written-protobuf wire snapshot (`MarshalSnapshot`/`UnmarshalSnapshot`, `snapshot.go`) both render, so the CLI and the v0.11 web console decode one typed model rather than re-parsing text. The cluster node builds its registry in `internal/cluster` (`initMetrics`: `build_info`/`node_info`, uptime, the cluster-wide gauges — members, voters, is-leader, effective generation — and the durability posture any node derives from its own replica — object-version and bucket counts, the active auto storage profile's k/m, layout-transition-open — plus the `hamster_s3_requests_total{method,code}` counter the `ServeS3` middleware increments) and serves the snapshot over the control channel (`reqMetrics`); `cmd/hamster` serves `/metrics`, renders `cluster metrics`, and prints a durability summary on `cluster status`. Request-latency histograms are the next observability increment.
- [`internal/sys`](internal/sys/) — production adapters over the OS. Thin and boring by rule: **no logic in the adapters** — code that makes a decision belongs on the core side of the seam.
- [`cmd/hamster`](cmd/hamster/) — the server binary. `hamster serve` is a single-node dev preview, but a durable one: blobs under `<data-dir>/blobs`, metadata in BadgerDB under `<data-dir>/meta` — restarts keep everything. `hamster cluster` (init, token, join, run, status, drain/undrain, remove, optimize, encrypt, recover; `join`/`run -token` take `-replaces <old>` to swap in for a member) manages a cluster node; `cluster run -s3 <addr>` serves the S3 API over the erasure-coded cluster data path (the v0.3 preview: leader-only writes, no multipart/copy yet); `cluster run -master-key-file <path>` loads the cluster KEK and `cluster encrypt` turns on encryption at rest (ADR-0021); `cluster run -new-master-key-file <path>` loads the incoming key and `cluster rotate-key` rewraps every object onto it ([ADR-0032](docs/adr/0032-master-key-rotation.md): metadata-only, the key never travels the wire, `cluster status` shows the rotation's progress); `cluster rotate-ca` rotates the cluster CA as a no-downtime dual-trust rollover ([ADR-0033](docs/adr/0033-ca-rotation.md): mint a new CA, trust it alongside the old, reissue every node onto it, drop the old — the new CA key never leaves the leader); `cluster can-stop <node>` is the upgrade interlock ([ADR-0034](docs/adr/0034-rolling-upgrade-machinery.md)). `cluster run -admin <addr>` (and `serve -admin <addr>`) serves the admin endpoints — Prometheus metrics at `/metrics` ([ADR-0035](docs/adr/0035-observability-metrics.md)), the operational surface the web console will later share — and `cluster metrics` prints a node's snapshot from the CLI.

## Development workflow

- The plan lives in the repo at two altitudes: high-level milestones per version in [docs/ROADMAP.md](docs/ROADMAP.md), and the current execution plan — what is being worked now and next, in priority order — in [docs/PLAN.md](docs/PLAN.md). PLAN.md is the front line only and is pruned as work lands (no growing TODO/kanban graveyard — a completed item's record survives in git history, its now-green test, and the shipped ADR/doc). Finer design detail still lives in each design doc's open-questions section; finishing a design means updating the affected docs (and writing a new ADR if a decision was made). GitHub Issues may take over once there are outside contributors.
- Keep pull requests small and focused: one issue, one concern.
- Sign every commit with `git commit -s` (Developer Certificate of Origin — see [ADR-0001](docs/adr/0001-apache-2-license-dco.md)).
- **Accepted ADRs are immutable.** Once an ADR is Accepted, do not edit its body. A changed or reversed decision is a *new* ADR that names exactly what it supersedes; the only edit permitted to the superseded one is its Status line — `Superseded by ADR-00X` or `Partially superseded by ADR-00X`, with a link. New consequences or rationale learned later also belong in a new ADR, never retrofitted into the old one. Fixing a typo or a dead link is fine; adding reasoning is not.
- **ADRs record decisions, not implementation status.** "Accepted" means the decision is made, not that the code exists yet — an ADR ahead of the code is normal and is *not* a superseded ADR. Never write "implemented / not yet implemented" into an ADR. Track decided-but-unbuilt work elsewhere: coarse milestones in [docs/ROADMAP.md](docs/ROADMAP.md); a specific behavioral gap as a skipped or failing test that asserts the ADR's behavior and names it (e.g. `TestClusterMetadataPersistsToBadgerDB`) — executable, self-closing, and unable to drift from reality.
- Keep docs and code in step in the same pull request. A design doc may be *corrected* when it states something factually wrong about the system, but progress tracking stays out of both ADRs and design-doc prose. Docs that contradict the code are worse than no docs.
- Keep the [README](README.md) version-status table current as releases move: the version under active work is `🚧 in progress`, shipped ones are ✅ (linked to their release), later ones `planned`. It is the user-facing status surface and must never lag the code.

## Naming

Use standard distributed systems vocabulary in code, docs, CLI, and logs: node, cluster, shard, partition, write buffer, data directory. Hamster is the brand, not an operational vocabulary — do not introduce themed names for system components.
